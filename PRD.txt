# PRD — Browser Multiplayer Monopoly Game Tanpa Giliran

## 1. Ringkasan Produk

Produk ini adalah game Monopoly berbasis browser yang dimainkan secara online oleh maksimal 4 pemain dalam satu room. Berbeda dari Monopoly klasik yang memakai sistem giliran, game ini menggunakan konsep **free-roll real-time**, yaitu setiap pemain dapat melempar dadu kapan saja saat memenuhi syarat tertentu, tanpa menunggu giliran pemain lain.

Tujuan utama produk adalah menciptakan pengalaman Monopoly yang lebih cepat, dinamis, kompetitif, dan cocok untuk permainan online singkat.

---

## 2. Tujuan Produk

### Tujuan Utama

* Membuat game Monopoly online multiplayer yang dapat dimainkan langsung dari browser.
* Mendukung 4 pemain dalam satu room.
* Menghilangkan sistem giliran klasik dan menggantinya dengan mekanisme lempar dadu bebas.
* Menjaga permainan tetap adil meskipun semua pemain bisa bergerak secara bersamaan.
* Memberikan pengalaman bermain yang cepat, interaktif, dan mudah dipahami.

### Tujuan Bisnis / Produk

* Menyediakan MVP game multiplayer casual berbasis web.
* Mendukung mode bermain cepat untuk teman, komunitas, atau publik.
* Menjadi fondasi untuk fitur lanjutan seperti ranking, skin token, custom board, dan turnamen.

## 5. Konsep Gameplay

### Konsep Dasar

Game menggunakan papan Monopoly dengan petak properti, pajak, kesempatan, dana umum, penjara, start, dan petak khusus lainnya. Setiap pemain memiliki token, uang awal, dan posisi di papan.

Perbedaan utama dari Monopoly klasik adalah tidak adanya sistem giliran. Semua pemain dapat melakukan lempar dadu secara mandiri ketika status pemain mengizinkan.

### Prinsip Utama Free-Roll

* Tidak ada urutan giliran pemain.
* Setiap pemain memiliki tombol “Roll Dice”.
* Pemain boleh melempar dadu jika tidak sedang dalam status terkunci.
* Setelah melempar dadu, pemain masuk ke fase resolusi aksi.
* Pemain harus menyelesaikan aksi pada petak tempat ia berhenti sebelum bisa melempar lagi.
* Untuk mencegah spam, setiap pemain memiliki cooldown setelah menyelesaikan satu roll.

## 8. Mekanisme Roll Dice Tanpa Giliran

### Syarat Pemain Bisa Roll

Pemain bisa menekan tombol Roll Dice jika:

* Game sudah dimulai.
* Pemain belum bangkrut.
* Pemain tidak sedang bergerak.
* Pemain tidak sedang menyelesaikan aksi tile.
* Pemain tidak sedang cooldown.
* Pemain tidak sedang dalam pop-up keputusan wajib.

### Cooldown Roll

Untuk menjaga keseimbangan permainan:

* Setelah satu aksi selesai, pemain masuk cooldown.
* Durasi cooldown MVP: 5 detik.
* Cooldown dapat disesuaikan di masa depan.

### Resolusi Roll

Saat pemain menekan Roll Dice:

1. Server menerima request roll.
2. Server memvalidasi status pemain.
3. Server menghasilkan angka dadu.
4. Server mengirim hasil roll ke semua pemain.
5. Token pemain bergerak di papan.
6. Server menentukan petak tujuan.
7. Server menjalankan efek petak.
8. Pemain menyelesaikan aksi jika diperlukan.
9. Pemain masuk cooldown.
10. Setelah cooldown selesai, pemain bisa roll lagi.

### Validasi Anti-Cheat

* Hasil dadu harus dibuat oleh server, bukan client.
* Client hanya mengirim request roll.
* Server menentukan angka, posisi, efek tile, pembayaran, pembelian, dan status pemain.
* Semua perubahan state game harus berasal dari server.

## 9. Tantangan Desain Karena Tidak Ada Giliran

Karena semua pemain dapat bergerak tanpa giliran, beberapa aturan klasik Monopoly perlu disesuaikan.

### Masalah 1: Banyak Pemain Berhenti di Properti yang Sama Secara Hampir Bersamaan

Solusi:

* Semua event diproses oleh server berdasarkan timestamp request yang valid.
* Jika dua pemain mendarat di properti kosong hampir bersamaan, pemain yang event-nya diproses lebih dulu mendapat hak beli lebih dulu.
* Pemain lain akan melihat status properti sudah berubah jika event pertama selesai lebih dulu.

### Masalah 2: Pemain Terlalu Cepat Mengumpulkan Uang / Properti

Solusi:

* Terapkan cooldown roll.
* Terapkan fase aksi wajib sebelum roll berikutnya.
* Batasi waktu keputusan pembelian properti.

### Masalah 3: Game Terlalu Kacau

Solusi:

* Gunakan animasi token yang jelas.
* Gunakan event log real-time.
* Gunakan notifikasi pembayaran/sewa.
* Tampilkan status setiap pemain: rolling, moving, deciding, cooldown.

### Masalah 4: Pemain Mengabaikan Keputusan Pembelian

Solusi:

* Setiap keputusan pembelian memiliki timer.
* Jika timer habis, sistem otomatis memilih “Skip Buy”.
* Untuk lelang, MVP dapat dinonaktifkan terlebih dahulu agar permainan lebih sederhana.

## 11. Properti

### Pembelian Properti

Jika pemain berhenti di properti kosong:

* Sistem menampilkan modal pembelian.
* Pemain dapat memilih Buy atau Skip.
* Timer keputusan MVP: 10 detik.
* Jika timer habis, sistem otomatis Skip.

### Membayar Sewa

Jika pemain berhenti di properti milik pemain lain:

* Server otomatis menghitung biaya sewa.
* Server otomatis memindahkan uang dari pemain yang mendarat ke owner.
* Jika saldo tidak cukup, pemain masuk proses bankruptcy.

### Properti Milik Sendiri

Jika pemain berhenti di properti miliknya sendiri:

* Tidak ada pembayaran.
* Pemain dapat lanjut ke cooldown.

### Upgrade Properti

Untuk MVP, upgrade rumah/hotel dapat dibuat sebagai fitur opsional.

Rekomendasi MVP:

* Tidak ada house/hotel dulu.
* Fokus pada loop roll, beli, bayar sewa, dan bangkrut.

Future:

* Pemain dapat membeli rumah/hotel dari panel properti.
* Upgrade dapat dilakukan kapan saja selama pemain tidak sedang menyelesaikan aksi wajib.

## 15. Kondisi Menang dan Kalah

### Kalah / Bankrupt

Pemain dianggap bangkrut jika:

* Saldo tidak cukup untuk membayar kewajiban.
* Pemain tidak memiliki aset yang bisa digunakan untuk membayar.

Untuk MVP sederhana:

* Jika saldo menjadi negatif setelah pembayaran, pemain langsung bankrupt.
* Semua properti milik pemain kembali ke bank.
* Pemain tidak bisa roll lagi.

### Menang

Game berakhir jika:

* Tersisa 1 pemain yang belum bangkrut.

Alternatif mode cepat:

* Game memiliki durasi maksimal, misalnya 20 menit.
* Jika waktu habis, pemain dengan total net worth tertinggi menang.

### Rekomendasi MVP

Gunakan dua mode kemenangan:

1. Last Player Standing.
2. Timed Match 20 menit.

## 17. Game State

### Server Authoritative State

Server menyimpan state utama game, meliputi:

* Room ID.
* Daftar pemain.
* Board state.
* Properti dan owner.
* Saldo pemain.
* Posisi pemain.
* Status pemain.
* Timer game.
* Event log.

Client hanya menampilkan state dan mengirim aksi pemain.

### Sinkronisasi State

* Saat pemain masuk room, server mengirim full game state.
* Saat ada aksi, server mengirim delta update.
* Jika reconnect, client meminta full state terbaru.

## 20. Non-Functional Requirements

### Performance

* Game harus berjalan lancar di browser modern.
* Latensi update real-time target di bawah 300ms untuk event normal.
* Animasi token tidak boleh menghambat validasi server.

### Scalability MVP

* Mendukung banyak room aktif secara paralel.
* Setiap room maksimal 4 pemain.
* State game dapat disimpan di memory untuk MVP.

### Reliability

* Jika pemain disconnect, status berubah menjadi disconnected.
* Pemain dapat reconnect ke room yang sama.
* Jika pemain disconnect terlalu lama, sistem dapat mengeluarkan pemain atau membuatnya inactive.

### Security

* Semua aksi harus divalidasi server.
* Client tidak boleh menentukan hasil dadu.
* Client tidak boleh mengubah saldo/properti secara langsung.
* Room code harus cukup sulit ditebak.

## 21. Reconnect dan Disconnect

### Disconnect

Jika pemain disconnect:

* Server menandai pemain sebagai disconnected.
* Token tetap berada di board.
* Pemain tidak dapat roll.
* Properti tetap dimiliki pemain.

### Reconnect

Jika pemain kembali:

* Pemain masuk menggunakan session/token lama.
* Server mengirim full game state terbaru.
* Pemain dapat lanjut bermain jika belum bankrupt.

### Timeout Disconnect

Rekomendasi MVP:

* Jika disconnect lebih dari 2 menit, pemain dianggap inactive.
* Dalam mode MVP, inactive player bisa otomatis bankrupt atau tetap diam sampai game selesai.

Rekomendasi: untuk MVP, gunakan status inactive dan jangan langsung bankrupt.

## 27. Risiko Produk dan Mitigasi

### Risiko: Permainan Terlalu Cepat dan Tidak Terkontrol

Mitigasi:

* Cooldown roll.
* Wajib menyelesaikan aksi sebelum roll lagi.
* Event log real-time.

### Risiko: Konflik State Karena Banyak Aksi Bersamaan

Mitigasi:

* Server authoritative state.
* Queue event per room.
* Semua transaksi game diproses secara atomik.

### Risiko: Pemain Bingung Karena Tidak Ada Giliran

Mitigasi:

* Status visual setiap pemain.
* Cooldown indicator.
* Highlight aksi penting.
* Tutorial singkat sebelum game.

### Risiko: Monopoly Klasik Tidak Cocok Tanpa Giliran

Mitigasi:

* MVP menggunakan rule sederhana.
* Beberapa fitur klasik seperti auction, trade, mortgage ditunda.
* Fokus pada fun loop cepat.
